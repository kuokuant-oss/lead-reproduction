# M4.4 handoff: dead-code and sampling semantics review

## Scope

M4.4 reviewed all tracked `StandardScaler` uses and the centralized
`downsample_indices` sampling pattern in `src/lead/` and `scripts/run_m3_*.py`.
No M5 modeling was started, and no BDG2 data was downloaded or processed.

## Decision

Keep both behaviors as intentional M3 reproduction-compatibility code:

+ `StandardScaler` remains in the M3 script fit path for numeric parity with the
  accepted original-script lineage, even though tree boosters are
  scale-invariant for split semantics.
+ `downsample_indices` keeps `[negs1, pos, negs2, pos]`, so positive rows appear
  exactly twice and the effective fit set remains 50:50.

ADR 0016 records the keep-vs-remove decision.

## Evidence

+ `src/lead/sample.py` documents the positive duplication behavior.
+ `src/lead/__init__.py` documents the package-level compatibility intent.
+ All tracked `StandardScaler` call sites in M3 scripts now carry a parity
  comment.
+ `tests/test_sampling_semantics.py` asserts that positives appear in both
  positive blocks, the sampled length is `4 * n_pos`, and the sampled label
  ratio is 50:50.
+ `data/processed/m4_4_sampling_scaler_decision.json` records the decision and
  source commit provenance.

M3.2 and M3.4 reruns were not required because labels, features, sampled
indices, and fit preprocessing remain byte-identical. Expected AUC delta is `0`,
within the `0.0005` regression noise floor in `tests/golden_metrics.json`.
