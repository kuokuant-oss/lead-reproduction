# Handoff: M4.2 Guarded Positional Label Alignment Complete

**Date**: 2026-06-24
**Status**: Complete - guard-only label integrity slice
**Scope**: M4.2 label assignment integrity; no key join was possible from raw files

## Summary

M4.2 confirmed that `data/raw/m3/bad_meter_readings.csv` has only the single
column `is_bad_meter_reading`. It does not contain `building_id`, `timestamp`,
or `meter`, so a true key-aligned label merge cannot be implemented without
fabricating keys.

The M3 label path therefore remains positional, but it is now guarded. Labels
stay byte-identical to the old assignment; the change is that invalid label
schema, length drift, non-canonical train row order, or undefined train row
identity fail before labels are assigned.

## Code Changes

`src/lead/data.py` now enforces:

+ `bad_meter_readings.csv` must contain exactly `is_bad_meter_reading`.
+ `len(bad_meter_readings) == len(train)`.
+ The train frame must retain the raw `RangeIndex` row order.
+ `(building_id, meter, timestamp)` must be unique in the train frame.

`tests/test_label_join_integrity.py` covers:

+ Successful byte-identical positional assignment.
+ Expected-column assertion.
+ Equal-length assertion.
+ Train row-key uniqueness assertion.
+ Synthetic row-order mismatch failure.
