# Handoff: M4.3 Timestamp Value-Change Regime Complete

**Date**: 2026-06-25
**Status**: Complete - timestamp-merge measured inside the M4 noise floor
**Scope**: M4.3 value-change feature regimes; no label, model, or BDG2 changes

## Summary

M4.3 adds two explicit value-change regimes to `add_value_change_features`:

+ `row_offset`: the default reproduction-compatible regime. This preserves the
  existing M3 `groupby("building_id").shift(n)` behavior.
+ `timestamp_merge`: an opt-in regime that computes n-hour offsets by joining
  timestamp-shifted readings. When `meter` is present, the merge key is
  `(building_id, meter, timestamp)` so the output remains one row per input row.

Offline and causal naming remains unchanged from ADR 0007: callers choose the
full `SHIFTS` set for offline past+future features or `PAST_SHIFTS` for causal
past-only features.

## Evidence

`scripts/run_m4_3_timestamp_value_change.py` measured both regimes in one M3.2
LightGBM 80/20 offline harness with the same split, seeds, downsampling, model,
and NaN treatment.

Row-offset produced AUC `0.9920119520500562`, Delta
`+0.000011952050056218688` versus golden `0.9920`.

Timestamp-merge produced AUC `0.9924831086743003`, Delta
`+0.00047115662424412896` versus row-offset.

The row-offset environment sanity anchor passed. The timestamp-merge regime
improved this run but stayed inside the +/- `0.0005` noise floor, so M4.3 keeps
`row_offset` as the default reproduction regime and exposes `timestamp_merge`
as an explicit opt-in regime.
